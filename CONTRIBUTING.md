# Contributing to gooq

Thank you for your interest in contributing to gooq, a type-safe SQL query
builder and code generator for Go, inspired by jOOQ. This guide describes how to
set up the project, run the test suite, regenerate generated code, and submit
changes.

## Prerequisites

- **Go 1.26 or newer.** The module targets the version declared in `go.mod`.
- **Docker.** Required only for the PostgreSQL integration tests. The integration
  suite starts a real PostgreSQL container; without Docker you can still run the
  remaining tests by passing the `-short` flag (see below).

## Repository layout

The repository is organized to keep the core query builder free of external
dependencies.

- **`gooq` (repository root)** — the core query builder package. It MUST remain
  dependency-free; `go.mod` contains no `require` directives.
- **`./codegen`** — the importable code generator library.
- **`./cmd/gooq-gen`** — the command-line interface that wraps `./codegen`.
- **`./internal/gen`** — generates the higher-arity `Record` and `Select` types,
  invoked through `go generate ./...`.
- **`./example`** — a runnable demonstration of the query builder.
- **`./integration`** — a **separate Go module** that holds the PostgreSQL
  integration tests. Keeping it in its own module ensures the core package never
  acquires test-only or driver dependencies.
  - **`./integration/internal/gendb`** — regenerates the integration accessors
    from the test schema.
  - **`./integration/testdata/schema.sql`** — the schema used by the integration
    tests.

## Building and testing

From the repository root:

```sh
go build ./...
go vet ./...
go test ./...
```

The integration module is built and tested separately, because it is a distinct
Go module:

```sh
cd integration
go test ./...
```

The integration tests require Docker. To skip the Docker-dependent tests, use the
`-short` flag:

```sh
cd integration
go test -short ./...
```

Run the full suite (core plus integration) before opening a pull request.

## Regenerating generated code

Some source files are produced by code generators and must be regenerated when
the underlying templates or schema change.

- **Arity files.** The higher-arity `Record` and `Select` types in
  `./internal/gen` are generated by:

  ```sh
  go generate ./...
  ```

  Run this from the repository root after modifying the relevant templates.

- **Integration accessors.** After editing
  `integration/testdata/schema.sql`, regenerate the integration accessors from
  within the integration module:

  ```sh
  cd integration
  go run ./internal/gendb
  ```

Always commit regenerated files together with the change that motivated them.

## Coding standards

- Format all code with `gofmt`. Code that is not gofmt-clean will not be merged.
- Provide Go doc comments on every exported identifier, written as complete
  sentences beginning with the identifier name.
- The core module MUST keep zero external dependencies. Do not add `require`
  directives to the root `go.mod`. Driver-specific or test-only dependencies
  belong in the `./integration` module.
- Write clear, self-documenting code and add comments only where the logic is not
  self-evident.

## Commit and pull request etiquette

- Write clear, descriptive commit messages that explain the intent of the change.
- Run the full test suite (core and integration) before submitting.
- If you changed generators, templates, or the integration schema, regenerate the
  affected code and include the regenerated files in your commit.
- Ensure `go build ./...`, `go vet ./...`, and `gofmt` all pass cleanly.
- Keep pull requests focused; describe what changed and why, and reference any
  related issues.

We appreciate your contributions.
